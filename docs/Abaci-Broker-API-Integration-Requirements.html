1. Executive Summary

Abaci Investments Limited operates as a BSEC-licensed Merchant Bank providing custodial and portfolio management services for clients trading on the Dhaka Stock Exchange (DSE) and Chittagong Stock Exchange (CSE). As a custodian and portfolio manager, Abaci does not execute trades directly — trade execution is the responsibility of our partner broker(s).

This document specifies the API integration requirements for connecting Abaci's back-office system with the broker's order management and execution infrastructure. The integration enables:

  • Order Routing: Abaci sends buy/sell orders on behalf of managed clients to the broker for execution on DSE/CSE
  • Execution Reports: Broker sends real-time/batch execution confirmations back to Abaci
  • Position & Settlement Data: Broker provides daily position statements and settlement reports for reconciliation
  • Reconciliation: Automated matching of Abaci's records (system of record) against broker's execution data
Key Principle: Abaci's back-office is the primary system of record for all client portfolios, holdings, and fund accounting. Broker data is consumed for execution confirmation and reconciliation purposes. Any discrepancy is flagged and resolved through the defined process.

2. Business Context & Relationship Model

2.1 Abaci's Role (Merchant Bank / Custodian / Portfolio Manager)

  • Manages client portfolios and makes investment decisions
  • Holds securities in custody via CDBL Depository Participant membership
  • Maintains the master record of client holdings, cash positions, and NAV
  • Initiates orders based on portfolio strategy and client mandates
  • Performs daily NAV calculation and client fund accounting
  • Handles corporate actions (dividends, rights, bonus) for custodial clients
  • Generates regulatory reports (BSEC, DSE, CDBL)

2.2 Broker's Role (Trade Execution)

  • Receives orders from Abaci and routes to DSE/CSE for execution
  • Manages exchange connectivity, order book, and execution quality
  • Sends execution reports (fills, partial fills, rejections) back to Abaci
  • Provides daily position/trade statements for reconciliation
  • Handles exchange-level settlement (T+2) with clearing house
  • Calculates and applies brokerage fees, exchange fees, regulatory charges

3.6 Market Data Feed OPTIONAL PHASE 2

If the broker provides market data via API, Abaci would consume:

  • End-of-day closing prices for NAV calculation
  • Intraday LTP for real-time portfolio valuation
  • Corporate action announcements (dividends, rights, bonus)
Note: Market data is optional from the broker. Abaci currently sources market data from DSE directly. However, if the broker can provide this, it simplifies the data pipeline and ensures price consistency for reconciliation.

5. Technical Requirements

5.1 Authentication & Security

RequirementSpecification
AuthenticationAPI Key + Secret (HMAC-SHA256) or OAuth 2.0 Client Credentials
TransportHTTPS/TLS 1.2+ mandatory. No plaintext HTTP.
IP WhitelistingAbaci will provide static IP(s) for whitelisting
Request SigningHMAC signature on request body to prevent tampering
IdempotencyAll POST endpoints must support idempotency keys to prevent duplicate orders
EncryptionSensitive fields (NID, account numbers) encrypted at rest

6.1 Daily Reconciliation Workflow

Reconciliation runs daily after market close and compares Abaci's records (system of record) against broker's data:

StepTimeActionOwner
114:30 BSTMarket closes. Broker finalizes trade data.Broker
215:00 BSTAbaci pulls/receives execution reports for the dayAbaci
315:30 BSTAbaci pulls broker's position statement for all clientsAbaci
416:00 BSTAutomated reconciliation runs: match Abaci holdings vs broker positionsAbaci
516:00 BSTDiscrepancy report generated (if any mismatches)Abaci
616:30 BSTDiscrepancies communicated to broker for investigationBoth
717:00 BSTNAV calculation finalized using reconciled dataAbaci
